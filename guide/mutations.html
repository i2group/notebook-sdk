    Changing data with mutations

    i2 Notebook Web API provides the ability to write data to the chart, either by adding new records or by editing or deleting existing records.

    In addition, the API allows modification of the selection state and the view state of the chart. Writing data is asynchronous, so all changes are made using "mutation" transactions.

    Mutations are transactions that are run as a single atomic action, containing one or more mutation operations that make changes to chart data or application state. This ensures that your changes cannot conflict with changes made by other plugins or the user of the application.

    The application will call your mutation code handler when specifically asked to start a mutation transaction.

    The handler will receive the following parameters:

    • A parameter of type IApplicationContents interface, providing read access to the schema, visual and record data for the chart, and also a payload implementing the IMutationsAPI
    • A parameter implementing one of the mutation interfaces, providing access to the available mutation operations that allow changes to be made to record and application state.

    Commit and rollback

    As with transactions in databases, once your mutation handler has finished, it must indicate whether the mutation operations created within it should be "committed" and applied to the chart and application, or to be "rolled back" and not applied.

    Committing a mutation will cause the mutation operations called in the handler to be applied to the chart. Some operations, such as selection and view changes, are applied immediately, while others, such as the writing of record data, are sent through the server before control is returned to the application.

    Mutation reports

    When committing or rolling back a mutation you can also provide information that is displayed to the user as a notification pop-up. See the ISuccessReport interface and 'IDetailedReport` interfaces that can be optionally supplied in the returned value from the tracked or untracked mutations handlers.

    Note that the fate of the transaction (i.e. commit or rollback) and the type of report presented to the user (success, failure, or nothing) are completely independent of each other. Reporting an error to the user is often associated with rolling back the mutation transaction, but you can combine a transaction commit with an error report, or a rollback with a success report.

    There are two standard design behaviors that i2 Notebook adheres to - demonstrated most obviously in the built-in Expand functionality - that are worth following in your own plug-ins to provide a helpful and unobtrusive user experience:

    1. It's not always necessary to provide an explicit success report to the user. When implementing a command that adds data to the chart, the success of the operation is often obvious from the fact that data has been added to the chart. Only provide an explicit 'success' report if there is something else of value to report to the user.

    2. If a command is supposed to add data to the chart but half-way through the mutation you discover that there isn't any (new) data to add, you should roll back the mutation and provide an explicit report to the user to indicate that there was nothing to add.

      If nothing were reported to the user at all in response to invoking the command, they might be unsure whether the command even ran, decreasing their confidence in the application and plug-in.

      The decision as to whether such a "nothing was added" report should be a 'success' or a 'failure' report will depend on factors such as whether the lack of new data is a normal situation or caused by an error.

    Tracked and untracked mutations

    i2 Notebook supports two different types of mutation:

    • "Tracked" mutations, created via the IApplication.runTrackedMutations method.

      Tracked mutations supply an ITrackedMutations object which provides a full set of chart mutation operations.

      A committed tracked mutation collects all of its mutation operations into a single entry in the application undo stack. The user can undo all of the chart changes from a tracked mutation with a single undo action.

    • "Untracked" mutations, created via the IApplication.runUntrackedMutations method

      Untracked mutations supply an IUntrackedMutations object which provides a more restricted set of chart mutation operations (typically chart record selection and chart view position and zoom level). These operations take place without creating an entry in the application undo stack.

    Commands which only affect selection or the view of the chart usually do not need to become user-undoable actions, so they are often implemented as untracked mutations. Other commands should be tracked so that the user can undo them.

    Cancellation and status messages

    If mutation handler is an asynchronous function, it may be canceled by use of a payload implementing the ICancelation interface. Triggering the cancelation will automatically roll back the mutation.

    While an asynchronous mutation handler is being run, the application may also show status messages provided by the mutations handler by calling the status object provided on either of the ITrackedMutations or IUntrackedMutations interfaces. See the IStatus interface.
